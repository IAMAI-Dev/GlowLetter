# GlowLetter 项目开发规范

## 1. 项目定位与依据

- 本项目是“绿荧来信”微信小程序，技术栈固定为原生 JavaScript、WXML、WXSS 与微信云开发 `wx.cloud`。
- 产品需求以 `docx/PRD.md` 为唯一基准；`opendesign/` 仅用于查阅交互、布局、视觉与文案，不作为生产代码。
- 当前处于 MVP 演示阶段。未确认的算法、标准曲线、污染阈值和处理建议不得自行推断，应记录到 `docs/OPEN_QUESTIONS.md`。

## 2. 目录与分层

```text
miniprogram/       小程序页面、组件、服务、适配器和公共工具
cloudfunctions/    登录、演示分析、检测记录和配置等云函数
docs/              架构、数据模型、开发计划与待确认问题
docx/              原始 PRD
opendesign/        OpenDesign 原型与设计参考，只读
```

- 页面只负责展示与交互；云能力统一经 `services/` 封装，页面不得直接访问云数据库。
- 图片来源通过 `ImageSourceAdapter.getImage()` 适配；分析能力统一通过 `analyzeDetection(input)` 调用。
- 演示分析与真实分析必须能通过适配器替换，页面主流程不得依赖具体算法实现。
- 根目录保留微信开发者工具配置；正式代码不得写入 `opendesign/`。

## 3. 开发边界

- 不引入 Taro、uni-app、Vue、React、自建服务器或 MySQL；不混用其他 CloudBase 接入方式。
- 第一版固定为 `demo` 模式。所有模拟结果必须保留 `isDemo: true`，并在界面显著标注“演示数据”。
- 不得使用“准确检测”“权威结论”等误导性表述，不得伪造真实科学公式或检测结论。
- 图片上传到云存储，数据库只保存文件 ID 与元数据，不保存 base64。
- 环境 ID、密钥和 API Key 不得硬编码或提交；私有数据读写必须由云函数校验当前用户身份和输入字段，不能信任前端传入的 `_openid` 或权限信息。

## 4. 代码与界面约定

- 文件与目录使用小写 kebab-case，JavaScript 变量和函数使用 camelCase，常量使用 UPPER_SNAKE_CASE。
- 公共文案、颜色、间距和状态值集中到配置或常量中，避免页面内重复硬编码。
- 可复用界面提取为组件；函数保持单一职责，只为业务原因和非显然逻辑添加注释。
- 项目明确使用自定义顶部导航和悬浮胶囊底部导航：所有页面设置 `navigationStyle: "custom"`，顶部组件必须适配状态栏与微信胶囊安全区域。
- `app.json` 不配置原生 `tabBar`；首页、历史、我的常驻在 `pages/home/home` 主页面壳层中，自定义底栏只切换本地视图，不触发页面路由。检测流程、结果和详情仍使用独立小程序页面。
- 页面路由统一通过 `utils/navigation.js` 执行，保持首帧背景和转场一致，避免直接调用路由 API 造成闪白或重复跳转。
- 带自定义底栏或固定操作区的页面必须预留底部安全区，保证最后一项内容不被遮挡。
- 完整处理 `draft`、`image_selected`、`analyzing`、`completed`、`failed` 状态，以及加载、空数据、失败和重试场景。
- 用户提示使用可理解的中文并给出下一步；堆栈和内部错误码仅进入开发日志。
- 视觉实现遵循 `opendesign/`：近黑、暖白、荧光绿为主；颜色集中为全局变量，等级信息同时提供文字，不能只靠颜色表达。

## 5. 工作流与验收

1. 开发前阅读 PRD、相关 OpenDesign 页面及现有代码，先明确本次范围。
2. 只实现已确认需求；需求缺失时更新 `docs/OPEN_QUESTIONS.md`，每完成一个阶段更新 `docs/DEVELOPMENT_PLAN.md`。
3. 提交前至少验证：微信开发者工具可编译、主流程可走通、页面返回正确、表单校验有效、加载/空/错状态完整。
4. 涉及云函数时验证输入校验、分页、删除权限、用户数据隔离，以及演示分析始终返回 `isDemo: true`。
5. 涉及界面或图片选择时进行真机检查；正式验收至少覆盖两种 Android 设备和一种 iOS 设备。
6. 每完成一个可独立验收的阶段性修改就创建一次提交；提交信息遵循 Conventional Commits，格式为 `<type>(可选 scope): 中文具体结果`，标题必须准确描述交付结果。
7. 若开发方向确认错误，先保护不相关的用户改动，再回到最近一个正确版本重新实现；不保留无意义的错误版本，也不创建 `revert` 类型提交，保持提交历史整洁。
