# GlowLetter 项目开发规范

## 1. 项目定位与依据

- 本项目是“绿荧来信”微信小程序，技术栈固定为原生 JavaScript、WXML、WXSS 与微信云开发 `wx.cloud`。
- 产品需求以 `docs/PRD.md` 为基准；`opendesign/` 仅用于查阅交互、布局、视觉与文案，不属于生产代码。
- 当前 `0.2.0` 原型演示版已经部署为体验版，云函数、数据库和存储链路可用于演示。真实算法、标准曲线、污染阈值和处理建议仍未确认，不得自行推断；缺失内容记录到 `docs/OPEN_QUESTIONS.md`。

## 2. 目录与分层

```text
miniprogram/       小程序页面、组件、样式、services、adapters 与公共工具
cloudfunctions/    登录、演示分析、检测记录和应用配置云函数
cloudbase/         不含敏感信息的种子数据、安全规则模板与配置说明
tests/             云函数和小程序服务自动化测试
docs/              PRD、架构、部署、开发计划与待确认问题
opendesign/        OpenDesign 原型和设计参考，只读且不提交
```

- 页面只负责展示与交互，云能力统一通过 `miniprogram/services/` 封装，页面不得直接访问云数据库。
- `adapters/` 隔离图片来源和分析实现；替换真实分析能力时不得让页面主流程依赖具体算法。
- `utils/store.js` 只保存草稿、页面间待展示结果和用户主动选择的离线演示记录。云端与离线历史不迁移、不混合。
- 根目录保留微信开发者工具共享配置；私有配置、环境标识和本机文件不得提交。

## 3. 开发边界

- 不引入 Taro、uni-app、Vue、React、自建服务器或 MySQL，不混用其他 CloudBase 接入方式。
- 当前版本固定为 `demo` 模式。所有模拟结果必须保留 `isDemo: true`，并在界面显著标注“演示数据”。
- 不得使用“准确检测”“权威结论”等误导性表述，不得伪造科学公式、检测结论或处理建议。
- 图片上传到云存储，数据库只保存文件 ID 与数值型元数据，不保存 base64。
- 环境 ID、密钥、API Key、OpenID、账号信息和本地绝对路径不得硬编码或提交。
- 私有数据读写必须由云函数校验当前 OpenID、输入字段与资源所有权，不能信任前端传入的 `_openid`、角色、状态或结果。

## 4. 代码与界面约定

- 文件与目录使用小写 kebab-case；JavaScript 变量和函数使用 camelCase；常量使用 UPPER_SNAKE_CASE。
- 公共文案、颜色、间距和状态值集中管理，避免页面内重复硬编码。
- 项目使用自定义顶部导航和悬浮胶囊底部导航；所有页面设置 `navigationStyle: "custom"`。
- `app.json` 不配置原生 `tabBar`。首页、历史、我的常驻在 `pages/home/home` 主页面壳层中，底栏只切换本地视图。
- 页面路由统一通过 `utils/navigation.js` 执行，保持首帧背景和转场一致。
- 完整处理 `draft`、`image_selected`、`analyzing`、`completed`、`failed` 状态，以及加载、空数据、失败和重试场景。
- 用户提示使用可理解的中文并给出下一步；堆栈和内部错误码只进入开发日志。
- 等级信息必须同时提供文字，不能只依靠颜色表达。

## 5. 云端安全约定

- 四个数据库集合只允许管理端读写，小程序端只调用云函数。
- 云函数调用规则要求已认证的小程序用户；函数内部继续使用 `cloud.getWXContext().OPENID` 做身份和所有权校验。
- 免费体验环境的云存储使用“仅创建者可读写”预设权限；不要为了自定义存储规则在联调阶段擅自升级套餐。
- 删除记录时先删除关联图片，再删除数据库记录；图片删除失败必须保留记录并提示重试，文件已不存在视为幂等成功。
- 正式发布前升级套餐、拆分开发与生产环境，并重新复核规则、配额、监控、备份和数据留存。

## 6. 工作流与验收

1. 开发前阅读 PRD、相关设计和现有代码，先明确本次范围。
2. 只实现已确认需求；需求缺失时更新 `docs/OPEN_QUESTIONS.md`，阶段完成后更新 `docs/DEVELOPMENT_PLAN.md`。
3. 提交前执行 JavaScript 语法检查、JSON 解析、自动化测试、`git diff --check` 和敏感信息扫描。
4. 涉及云函数时验证输入校验、分页、删除权限、用户数据隔离，以及演示分析始终返回 `isDemo: true`。
5. 涉及界面或图片选择时进行真机检查；正式验收至少覆盖两种 Android 设备和一种 iOS 设备。
6. 提交信息遵循 Conventional Commits，格式为 `<type>(可选 scope): 中文具体结果`。
7. 发现方向错误时保护无关改动并回到最近的正确实现，不使用破坏性 Git 命令清理他人工作。
